Inugami-NG : InuCacheServices

In many cases, it is necessary to store values in a frontend-side cache to limit calls to the backend. The retrieved values can have varying retention periods.

There are two main use cases:

  • Session-based values, which are linked to the current session and must be cleared upon a page refresh (browser refresh).
  • Slow-changing backend values (such as reference data, labels, etc.). These values must be retained for a much longer duration than a single user session.

Managing a frontend-side cache also requires careful attention to potential collisions between applications running on the same domain name.

Configuration

The cache requires specific application details, such as its environment, name, and version. In the Angular application configuration, a new provider must be defined to carry this information. This is achieved using the provideCacheTracking method, which leverages signals to provide these details.

Using signals allows these values to be redefined after fetching the information from the server, without blocking the application's execution.

Usage

When using the cache, there are different scenarios to handle:

  • Retrieving a value from the cache.
  • Handling concurrent requests, where other actions might already be fetching the same value from the server.
  • Defining (setting) a value in the cache.
  • Clear an item from the cache

This is the simplest case, where we want to retrieve a value from the cache. To do this, if the value is session-related, we use the get method; if it has a specific retention policy, we use the getTTL method.

The InuCacheServices service internally handles potential collisions between applications. The provided key should be a functional key, without needing to account for this issue.

In more complex scenarios where multiple components might trigger the same request, it is crucial to handle pending states. To do this, you can use the getPending and getPendingTTL methods, which return an Observable to retrieve the value once the initial call is completed.

For the pending mechanism to work, the first call must invoke the setPending method to provide the observable to the cache. This allows the cache to notify all subsequent calls once the result is available.

Once our service has retrieved the data from the server, it can be added to the cache.

Depending on your application's logic, it may be necessary to evict values from the cache. To handle this, the clearKey, clearKeyTTL, and clear methods are available.
